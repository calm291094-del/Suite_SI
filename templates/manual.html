Manual de Administradores

Suite de Seguridad Informática SuiteSeg.cu — Versión 1.0

Este manual proporciona una guía completa y detallada para la administración de la Suite de Seguridad Informática, un sistema desarrollado para cumplir con las normativas vigentes en la República de Cuba en materia de seguridad de la información, gestión de activos de TI y control de acceso a recursos informáticos.

Audiencia: Este documento está dirigido a los administradores del sistema (Root, Administrador de Red y Responsable de Seguridad Informática). Los usuarios normales deben consultar la guía rápida que se les proporciona al registrar su cuenta.

Instalación Paso a Paso

Paso 1: Preparar el entorno

# Crear carpeta del proyecto
mkdir suite_seguridad && cd suite_seguridad

# Crear y activar entorno virtual
python -m venv venv
source venv/bin/activate    # Linux/Mac
venv\Scripts\activate             # Windows

Paso 2: Instalar dependencias

pip install django==4.2.7 reportlab==4.0.7 Pillow==10.1.0 django-widget-tweaks==1.5.0

Paso 3: Copiar los archivos del proyecto

Copie todos los archivos según la estructura de carpetas indicada en la documentación del código fuente. Asegúrese de que todos los archivos __init__.py existan en cada paquete Python.

Paso 4: Inicializar la base de datos

# Crear las migraciones (tablas en la BD)
python manage.py makemigrations

# Ejecutar las migraciones
python manage.py migrate

Paso 5: Crear datos iniciales

# Crear departamentos predefinidos
python manage.py crear_datos_iniciales

# Crear el super usuario ROOT
python manage.py crear_root
# Usuario: root | Contraseña: **********

Paso 6: Iniciar el servidor

python manage.py runserver
# Acceder en: http://127.0.0.1:8000/

Listo: El sistema está operativo. Inicie sesión con root / ********** para acceder como super administrador.

Paso 7 (Producción): Despliegue con Gunicorn + Nginx

# Instalar Gunicorn
pip install gunicorn

# Iniciar con Gunicorn (4 workers)
gunicorn suite_seguridad.wsgi:application --bind 0.0.0.0:8000 --workers 4

# Configurar Nginx como proxy inverso
# Archivo: /etc/nginx/sites-available/suiteseg
server {
    listen 80;
    server_name suiteseg.institucion.cu;
    
    location /static/ { alias /ruta/a/staticfiles/; }
    location /media/  { alias /ruta/a/media/; }
    
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

Root (Super Administrador)

Rol máximo e irremovible. Solo existe un root en el sistema. Tiene acceso absoluto a todas las funcionalidades. Es el único que puede asignar roles de administrador (Admin de Red y Seguridad Informática). No puede ser eliminado, desactivado ni degradado por ningún otro usuario, ni por sí mismo.

  • Credenciales por defecto: Usuario root, Contraseña **********
  • Primera acción obligatoria: Cambiar la contraseña inmediatamente después del primer acceso
  • Puede acceder al panel de administración de Django en /admin/
  • Único que puede: asignar roles admin_red y seguridad

Seguridad crítica: Si se pierde la contraseña del root, no hay recuperación directa. Se debe usar el comando python manage.py cambiar_password_root (o crear un nuevo superusuario desde la línea de comandos con python manage.py changepassword root).

Administrador de Red

Rol administrativo. Gestiona la infraestructura de red y el inventario físico de equipos. Puede gestionar usuarios (activar/desactivar/eliminar, pero no puede asignar roles admin). Tiene acceso completo al inventario y a los reportes PDF.

Responsabilidades según la base legal:

  1. Mantener actualizado el inventario detallado de todos los equipos conectados a la red
  2. Registrar cada componente de hardware (CPU, RAM, disco, etc.) para cada equipo
  3. Vigilar que ningún equipo no autorizado se conecte a la red institucional
  4. Documentar la topología de red y cualquier cambio realizado
  5. Monitorear el tráfico de red y detectar anomalías
  6. Mantener actualizados los firmwares de routers, switches y access points
  7. Gestionar las direcciones IP, VLANs y configuración DNS/DHCP
  8. Implementar y verificar respaldos de la configuración de los dispositivos de red
  9. Responder ante caídas de red minimizando el tiempo de inactividad
  10. Reportar al Responsable de Seguridad cualquier incidente detectado

Responsable de Seguridad Informática

Rol administrativo. Gestiona las políticas de seguridad, investiga incidentes, realiza auditorías y verifica el cumplimiento normativo. Comparte permisos de gestión de usuarios e inventario con el Admin de Red.

Responsabilidades según la base legal:

  1. Elaborar, actualizar y difundir las políticas de seguridad de la información
  2. Realizar evaluaciones de riesgo periódicas (mínimo anual, Res. 105/2020)
  3. Gestionar el Sistema de Gestión de Seguridad de la Información (SGSI)
  4. Investigar y documentar todos los incidentes de seguridad informática
  5. Clasificar la información: pública, interna, confidencial, secreta
  6. Verificar el cumplimiento de las contraseñas (mínimo 8 caracteres, complejidad)
  7. Supervisar la gestión de accesos lógicos a sistemas y bases de datos
  8. Planificar y ejecutar auditorías de seguridad internas
  9. Capacitar al personal en buenas prácticas de seguridad informática
  10. Mantener actualizado el plan de respuesta a incidentes
  11. Elaborar informes periódicos para la alta dirección
  12. Gestionar las relaciones con entidades de ciberseguridad nacional (CERT, MIC)

Usuario Normal

Rol básico. Puede usar el dashboard, acceder a documentos según su departamento, usar el chat interno, ver la base legal, efemérides y enlaces. No ve ningún panel administrativo. Si intenta acceder a una URL protegida (como /admin/usuarios/), el sistema lo redirige al login con un mensaje de error.

Gestión de Usuarios

Registro de nuevos usuarios

Cualquier persona puede registrarse desde la página pública llenando el formulario con:

  • Nombre completo: Nombre y apellidos reales (se usan en el sistema)
  • Nombre de usuario: Identificador único para iniciar sesión
  • Cargo: Puesto de trabajo en la institución
  • Departamento: Seleccionado de una lista predefinida
  • Contraseña: Mínimo 8 caracteres, con mayúsculas, minúsculas, números y caracteres especiales
  • Confirmar contraseña: Debe coincidir exactamente

Al registrarse, el usuario recibe automáticamente el rol Usuario Normal. Los administradores pueden cambiar su rol posteriormente.

Notificación automática: Cuando un usuario se registra, todos los administradores (Root, Admin de Red, Seguridad) reciben una notificación automática informándoles del nuevo registro, para que puedan evaluar y asignar el rol adecuado.

Indicador de fortaleza de contraseña

Durante el registro, el sistema muestra en tiempo real un indicador de fortaleza con:

  • Barra de progreso que cambia de color según la fortaleza
  • Porcentaje numérico de 0% a 100%
  • Texto descriptivo: Muy débil, Débil, Aceptable, Buena, Excelente

El cálculo evalúa: longitud, mayúsculas, minúsculas, números, caracteres especiales, longitud extra (>12, >16), mezcla de tipos, caracteres repetidos y secuencias comunes.

Ojito de visibilidad de contraseña

Tanto en el login como en el registro, cada campo de contraseña tiene un botón con un ícono de ojo () que permite alternar entre mostrar y ocultar el texto de la contraseña. Esto ayuda al usuario a verificar que escribió correctamente la contraseña sin exponerla permanentemente.

Panel de administración de usuarios

Accesible desde /admin/usuarios/ solo para roles administrativos. Muestra una tabla con todos los usuarios del sistema con las siguientes acciones:

Cambiar rol

Un selector desplegable junto a cada usuario permite cambiar su rol. Restricciones:

  • Root no puede ser modificado — el selector no aparece para root
  • Solo Root puede asignar los roles admin_red y seguridad
  • Cualquier admin puede cambiar un rol admin a usuario
  • Cualquier admin puede cambiar un usuario a usuario (sin efecto si ya lo es)

Activar / Desactivar

Un botón permite desactivar una cuenta. El usuario desactivado no podrá iniciar sesión, pero sus datos se conservan en el sistema. Restricciones:

  • Root nunca puede ser desactivado
  • Un admin no puede desactivarse a sí mismo

Eliminar usuario

Elimina permanentemente al usuario, su perfil, mensajes de chat y permisos. Los documentos que subió se conservan (se asignan a "Sistema"). Restricciones:

  • Root NUNCA puede ser eliminado — está protegido con triple verificación en el código
  • Un admin no puede eliminarse a sí mismo
  • Se muestra un modal de confirmación antes de ejecutar la eliminación

Protección de Root: La protección del usuario root está implementada en tres niveles: (1) el botón no aparece en la interfaz, (2) la API rechaza la solicitud si detecta que el target es root, (3) se verifica el nombre de usuario como última línea de defensa. Incluso si alguien modifica el HTML o envía una petición directa a la API, el servidor siempre rechazará la acción.

Encriptación de contraseñas

Django utiliza el algoritmo PBKDF2 con SHA-256 por defecto para encriptar todas las contraseñas. Esto significa que:

  • Las contraseñas nunca se almacenan en texto plano en la base de datos
  • Incluso si alguien accede al archivo de la BD, no podrá leer las contraseñas
  • Cada contraseña tiene un salt aleatorio único que previene ataques rainbow table
  • El número de iteraciones es de 390,000 por defecto (configurable en settings)

En la BD, las contraseñas se ven así: pbkdf2_sha256$390000$salt aleatorio$hash resultado

El inventario permite filtrar por:

Los filtros se combinan (búsqueda AND): si selecciona departamento "DI" y estado "activo", solo mostrará equipos activos de Dirección de Informática.

Monitoreo de Cambios de Hardware

Este es uno de los módulos más críticos del sistema. Permite detectar, registrar y notificar cualquier cambio realizado en los componentes de un equipo. Cumple con la Resolución 56/2019 sobre control de activos de información.

Cómo funciona

1

Registro inicial: Cuando se agrega un componente, se guarda una "huella digital" textual con todos sus datos (tipo, marca, modelo, especificaciones, capacidad, número de serie, estado) en el historial.

2

Detección de cambios: Cuando un administrador edita un componente existente, el sistema compara automáticamente la huella anterior con la nueva. Si detecta cualquier diferencia, procede al paso 3.

3

Registro en historial: Se crea un registro en HistorialComponente con: valor anterior, valor nuevo, campos que cambiaron, quién lo hizo y cuándo.

4

Notificación automática: Se envía una notificación a todos los administradores (excepto al que hizo el cambio) informando del cambio detectado, con detalles del equipo, componente, quién lo modificó y enlace directo.

Historial de cambios

En la página de edición de cada equipo, la sección "Historial de Cambios" muestra:

  • Iconos de color: verde para creaciones, amarillo para modificaciones, rojo para eliminaciones
  • Tipo de acción, componente afectado y usuario que realizó el cambio
  • Fecha y hora exacta del cambio
  • Detalle expandible con "Ver cambios" que muestra el estado anterior vs el nuevo

Limitación actual: El monitoreo actual es manual: detecta cambios cuando un administrador edita un componente en el sistema. No es un monitoreo automático del hardware en tiempo real. Para eso se necesitaría un agente instalado en cada equipo (ver sección de Ideas de Mejora).

FTP de Documentos

Sistema de compartir documentos internos con control de acceso granular por departamento. Funciona como un FTP interno donde los usuarios pueden subir y descargar archivos según sus permisos.

Subir un documento

Desde /documentos/subir/, cualquier usuario logueado puede:

  • Seleccionar un archivo (máximo 50 MB)
  • Agregar una descripción del contenido
  • Elegir la visibilidad: todos o departamentos específicos

Control de acceso por departamento

Si se desmarca "Visible para todos", aparece una lista de checkboxes con todos los departamentos. Se pueden seleccionar varios. El documento solo será visible para:

  • Usuarios pertenecientes a los departamentos seleccionados
  • Todos los administradores (siempre tienen acceso completo)

Notificaciones al subir

Al subir un documento:

  • Si es público: se notifica a todos los usuarios del sistema
  • Si es restringido: se notifica solo a los usuarios de los departamentos seleccionados

Descargar y eliminar

  • Descargar: Cualquier usuario con permiso puede descargar. Se registra el contador de descargas.
  • Eliminar: Solo el usuario que subió el documento o un administrador puede eliminarlo. Se muestra un modal de confirmación.

Formatos permitidos

PDF, DOC, DOCX, XLS, XLSX, PPT, PPTX, ZIP, RAR, TXT, JPG, PNG. El sistema detecta automáticamente el tipo MIME del archivo.

Chat Interno

Sistema de mensajería interna para comunicación en tiempo real entre usuarios del sistema. Soporta comunicación grupal (canal general) y privada (1 a 1).

Canal General

Todos los usuarios logueados pueden ver y enviar mensajes al canal general. Es útil para:

  • Avisos generales del administrador
  • Consultas técnicas rápidas
  • Comunicación de incidencias

Chats Privados

En la lista de contactos (panel izquierdo), al hacer clic en un usuario se abre un chat privado. Los mensajes privados solo pueden verlos los dos participantes.

Los canales privados se identifican como privado_X_Y donde X e Y son los IDs de los usuarios ordenados de menor a mayor, garantizando que ambos usuarios compartan el mismo canal.

Técnica de polling

El chat usa AJAX polling cada 3 segundos: el navegador consulta al servidor si hay mensajes nuevos después del último timestamp recibido. Si no hay mensajes nuevos, la respuesta es vacía (muy ligera). Ventajas: simple de implementar, compatible con cualquier navegador, no requiere WebSockets.

Seguridad del chat

  • Solo usuarios autenticados pueden acceder
  • Los mensajes se validan (máximo 1000 caracteres)
  • Se escapa el HTML para prevenir ataques XSS
  • Los mensajes privados se verifican: no se puede enviar a un canal privado sin ser uno de los participantes

Reportes PDF

El sistema puede generar un reporte completo del inventario en formato PDF profesional, listo para imprimir o archivar digitalmente.

Contenido del PDF

  • Encabezado: título, fecha de generación, referencia a la Resolución 127/2020
  • Tabla resumen: todos los equipos con ID, tipo, marca, ubicación, departamento, IP y estado
  • Detalle por equipo: para cada equipo, tabla de todos sus componentes con tipo, marca, modelo, especificaciones, capacidad y número de serie
  • Pie de página: nota de propiedad y amparo legal

Generación

Desde /admin/inventario/, el botón "Descargar PDF" genera el inventario completo. También se puede filtrar por departamento antes de generar, usando el filtro de departamento y luego el enlace de PDF.

El PDF se genera en el servidor usando la librería ReportLab, no requiere conexión a internet ni servicios externos. Formato A4 con márgenes profesionales.

Seguridad del Sistema

Protección de URLs

Todas las vistas internas están protegidas por decoradores personalizados:

  • @login_requerido: redirige al login si no está autenticado
  • @admin_requerido: verifica rol admin, retorna 403 Forbidden si no lo tiene
  • @root_requerido: verifica rol root exclusivamente

Un usuario malintencionado que conozca la URL /admin/usuarios/ no podrá acceder si no tiene rol administrativo. El servidor rechaza la solicitud antes de procesarla.

Protección CSRF

Django incluye protección CSRF (Cross-Site Request Forgery) en todos los formularios POST. Cada formulario incluye un token oculto que el servidor valida. Las llamadas AJAX envían el token en el header X-CSRFToken.

Protección XSS

Django escapa automáticamente las variables en los templates ({% templatetag openblock %} autoescape on {% templatetag closeblock %} es el comportamiento por defecto). En el chat, se usa escapeHtml() en JavaScript para escapar el contenido de los mensajes antes de insertarlos en el DOM.

Headers de seguridad (producción)

En settings.py hay líneas comentadas que deben activarse en producción:

# Descomentar estas líneas cuando DEBUG = False
SECURE_SSL_REDIRECT = True        # Forzar HTTPS
SESSION_COOKIE_SECURE = True       # Cookies solo por HTTPS
CSRF_COOKIE_SECURE = True          # Token CSRF solo por HTTPS
SECURE_BROWSER_XSS_FILTER = True   # Filtro XSS del navegador
SECURE_CONTENT_TYPE_NOSNIFF = True # Prevenir MIME sniffing
X_FRAME_OPTIONS = 'DENY'           # Prevenir clickjacking

Políticas de contraseñas

Las contraseñas cumplen con la Resolución 127/2020 (Art. 12):

  • Mínimo 8 caracteres
  • Al menos una letra mayúscula
  • Al menos una letra minúscula
  • Al menos un número
  • Al menos un carácter especial (!@#$%^&*()_+-=[]{}|;:,.<>?)
  • El indicador de fortaleza muestra el porcentaje en tiempo real
  • No se permiten contraseñas comunes (validación de Django)
  • No se permite que la contraseña sea similar al nombre de usuario

Solución de Problemas

Error: "No se pudo importar Django"

pip install django==4.2.7
# Verificar instalación
python -c "import django; print(django.VERSION)"

Error: "No module named 'core'"

Verifique que todos los archivos __init__.py existan en las carpetas suite_seguridad/ y core/.

Error: "table already exists" al migrar

# Borrar la BD y recrear
rm db.sqlite3
python manage.py makemigrations
python manage.py migrate
python manage.py crear_datos_iniciales
python manage.py crear_root

No carga el CSS / imágenes

# En desarrollo, verificar que STATIC_URL esté correcto en settings.py
# Ejecutar collectstatic para producción
python manage.py collectstatic

Error: "CSRF verification failed"

Esto ocurre si la sesión expiró. El usuario debe recargar la página e intentar nuevamente. Para sesiones más largas, ajustar en settings.py:

SESSION_COOKIE_AGE = 86400  # 24 horas (por defecto es 2 semanas)

Error: "Password reset" para root

python manage.py changepassword root
# O desde el shell de Python
python manage.py shell
>>> from django.contrib.auth.models import User
>>> root = User.objects.get(username='root')
>>> root.set_password('nueva_contraseña_segura')
>>> root.save()

Los documentos no se pueden subir (error 413)

Si usa Nginx, agregar en la configuración:

client_max_body_size 50M;

El chat no muestra mensajes nuevos

Verifique la consola del navegador (F12 → Console) por errores de JavaScript. El polling requiere que el usuario esté en la página del chat. Si la pestaña está inactiva, algunos navegadores reducen la frecuencia del setInterval.

Ideas de Mejora

A continuación se proponen mejoras organizadas por prioridad y complejidad de implementación. Cada idea incluye una justificación basada en la base legal o en necesidades prácticas detectadas.

Prioridad Alta — Impacto directo en la seguridad

Historial de sesiones de usuario

ALTA

Registrar cada inicio de sesión: fecha, hora, dirección IP, navegador y dispositivo. Permitir al usuario ver sus propias sesiones y al admin ver las de todos. Base legal: Res. 128/2020 Art. 8 (controles de acceso — registro de accesos).

Expiración y rotación de contraseñas

ALTA

Obligar a los usuarios a cambiar su contraseña cada N días (configurable por rol). Mostrar un aviso cuando falten X días para el vencimiento. Bloquear la cuenta si no se cambia a tiempo. Base legal: Res. 128/2020 Art. 12 (gestión de contraseñas — periodicidad de cambio).

Bloqueo de cuenta por intentos fallidos

ALTA

Bloquear la cuenta después de 5 intentos fallidos de login durante 30 minutos. Notificar al administrador. Base legal: Res. 56/2019 Art. 4 (autenticación — protección contra fuerza bruta).

Autenticación de dos factores (2FA)

ALTA

Implementar TOTP (Time-based One-Time Password) con apps como Google Authenticator. Obligatorio para roles administrativos, opcional para usuarios normales. Base legal: Res. 56/2019 Art. 4 (autenticación multifactor cuando sea posible).

Registro de auditoría completo

ALTA

Crear un modelo de auditoría que registre TODAS las acciones del sistema: quién hizo qué, cuándo, desde qué IP. Incluyendo: login/logout, cambios de roles, subida/borrado de documentos, cambios de inventario, etc. Los logs deben ser inmutables (no editables ni borrables). Base legal: Res. 56/2019 Art. 16 (registros de eventos de seguridad con retención mínima de 6 meses).

Prioridad Media — Funcionalidades prácticas

Agente de monitoreo automático de hardware

MEDIA

Desarrollar un script (Python) que se instale en cada equipo y envíe periódicamente al servidor los datos de hardware (CPU, RAM, disco, MAC). El servidor compara con el inventario registrado y genera alertas si detecta diferencias. Esto convierte el monitoreo de manual a automático. Protocolo: API REST con autenticación por token.

Dashboard de estadísticas avanzadas

MEDIA

Agregar gráficos: equipos por departamento, documentos subidos por mes, actividad del chat, incidentes de seguridad por tipo. Usar Chart.js o ApexCharts. Mostrar en el dashboard de admins un resumen visual del estado de la seguridad.

Mapa de red interactivo

MEDIA

Visualización gráfica de la topología de red con los equipos registrados. Permite ver qué equipos están en qué VLAN, switches, routers. Herramientas: D3.js o vis.js. Base legal: Res. 128/2020 Art. 18 (documentar la topología de red).

Sistema de tickets de soporte

MEDIA

Permitir que los usuarios creen tickets de soporte técnico (problemas de red, software, hardware). Los admins los asignan, responden y cierran. Incluye prioridad, estado, historial de la conversación y satisfacción del usuario.

Módulo de capacitación y evaluaciones

MEDIA

Crear cursos cortos sobre seguridad informática con cuestionarios. Llevar registro de quién se ha capacitado y su calificación. Base legal: Res. 128/2020 Art. 30 (formación y concienciación del personal en seguridad informática).

Análisis de vulnerabilidades integrado

MEDIA

Interfaz para ejecutar y documentar análisis de vulnerabilidades: escaneo de puertos, verificación de actualizaciones, revisión de configuraciones. Guardar los resultados en el sistema con plan de acción para cada vulnerabilidad encontrada.

Plan de respuesta a incidentes digital

MEDIA

Crear y mantener el plan de respuesta a incidentes dentro del sistema, con plantillas por tipo de incidente, flujos de escalado, y registro de cada incidente con lecciones aprendidas. Base legal: Res. 128/2020 Art. 20 (gestión de incidentes).

Gestión de políticas con vencimiento

MEDIA

Crear documentos de políticas de seguridad dentro del sistema con fecha de vigencia. Notificar cuando una política esté próxima a vencer y requerir su revisión. Base legal: Res. 105/2020 (revisión periódica del SGSI).

Prioridad Baja — Mejoras de experiencia

Modo oscuro / claro

BAJA

Permitir al usuario alternar entre el tema oscuro actual (cyber-noir) y un tema claro para uso diurno. Guardar la preferencia en localStorage.

Multiidioma (es/en)

BAJA

Implementar internacionalización con el sistema i18n de Django para permitir la interfaz en español e inglés.

Preferencias de notificaciones

BAJA

Permitir al usuario configurar qué tipos de notificaciones desea recibir y por qué canal (dentro del sistema, email si se configura).

Personalización de avatar

BAJA

Permitir a los usuarios subir una foto de perfil o seleccionar un avatar. Mejora la identificación visual en el chat y en los paneles administrativos.

Exportar datos a Excel/CSV

BAJA

Además del PDF, permitir exportar inventarios, listas de usuarios y documentos a formatos Excel (.xlsx) y CSV para análisis en otras herramientas.

Búsqueda global

BAJA

Barra de búsqueda global en el dashboard que busque en documentos, equipos, usuarios y mensajes de chat simultáneamente, tipo "Google interno".

Visionario a futuro — Requiere mayor planificación

Monitoreo de red en tiempo real

FUTURO

Integrar con herramientas como Zabbix, Prometheus o Grafana para mostrar en el dashboard estado de switches, routers, ancho de banda, latencia y alertas de red en tiempo real.

Integración con sistema de tickets externo

FUTURO

Conectar con GLPI o OTRS para sincronizar tickets de soporte, inventario de equipos y gestión de licencias de software.

Laboratorio de pruebas de seguridad (sandbox)

FUTURO

Sección donde los admins puedan simular ataques (phishing, fuerza bruta, SQLi) contra equipos controlados y documentar los resultados para capacitación.

IA para detección de anomalías

FUTURO

Modelo de machine learning que analice los logs de acceso y el tráfico de red para detectar patrones anómalos automáticamente: accesos fuera de horario, descargas masivas, cambios frecuentes de hardware, etc.

App móvil (PWA)

FUTURO

Convertir la suite en una Progressive Web App para que los admins puedan recibir notificaciones push, consultar el inventario y gestionar incidentes desde el celular, incluso sin conexión completa (cache offline).

Notificaciones por correo electrónico

FUTURO

Configurar un servidor SMTP (puede ser el correo institucional) para enviar notificaciones críticas por email: cambios de hardware, nuevos usuarios, alertas de seguridad. Así los admins son notificados incluso si no están en la suite.

Control de versiones de documentos

FUTURO

Implementar versionado de documentos: si alguien sube una nueva versión de un archivo existente, conservar las versiones anteriores con historial de cambios, quién subió cada versión y cuándo. Tipo "Git para documentos".

Gamificación de la seguridad

FUTURO

Sistema de puntos y logros por completar capacitaciones, detectar phishing simulado, reportar vulnerabilidades, etc. Fomenta la cultura de seguridad entre los usuarios. Tabla de clasificación por departamento.

Matriz de riesgos interactiva

FUTURO

Herramienta visual para crear y mantener la matriz de riesgos de la información: identificar activos, amenazas, vulnerabilidades, probabilidad, impacto y controles mitigadores. Cumple con la Res. 105/2020 (evaluación de riesgos del SGSI). Generar gráficos de calor del riesgo residual vs inherente.

API REST completa y documentación

FUTURO

Usar Django REST Framework para exponer una API completa del sistema con documentación Swagger/OpenAPI. Permite integraciones con otros sistemas institucionales: sistemas de gestión humana, control de acceso físico, etc.

Consejo de implementación: Priorice las mejoras de "Prioridad Alta" primero, ya que tienen impacto directo en el cumplimiento normativo. Luego avance con las de "Prioridad Media" según las necesidades de su institución. Las de "Futuro" son visiones a mediano/largo plazo que requieren planificación y presupuesto.
